Documento para: Equipo Augetec

Propósito: Compartir el contexto del proyecto, los resultados de las pruebas realizadas sobre la API de TecDoc, la arquitectura deseada y las ideas de funcionalidad, para que Augetec nos proponga un plan y scope de trabajo.

1. Contexto del negocio

  • Embler vende autopartes para autos europeos (BMW, Mercedes-Benz, Audi, VW, Porsche, Mini, etc.) en México.
  • La tienda en línea corre sobre Shopify, con un catálogo de ~12,700 productos ya publicado.
  • El ERP de la operación es Odoo (en implementación/operación con Augetec).
  • Contratamos/evaluamos TecDoc (TecAlliance), el catálogo estándar mundial de autopartes, que aporta dos cosas:
    • Datos de artículos: números OEM, especificaciones técnicas, códigos de barras, categorías estandarizadas, imágenes — todo disponible en español.
    • Datos de vehículos y compatibilidades (vínculo pieza ↔ vehículo): la base para un buscador por vehículo (marca → modelo → motorización → piezas que le quedan).

2. Objetivo principal del proyecto

Que el cliente final pueda encontrar la pieza correcta para su auto. Dos modos de búsqueda:

  1. Buscador por vehículo: el cliente selecciona Marca → Modelo/Serie → Año/Motorización y ve únicamente las piezas compatibles. Es el modo principal y el estándar en e-commerce de autopartes.
  2. Buscador por número de parte / OEM: el cliente (mecánico o conocedor) pega un número de parte original o de cualquier marca y encuentra el equivalente disponible en Embler.

Adicionalmente, usar TecDoc para enriquecer el catálogo: completar números OEM faltantes, especificaciones técnicas, compatibilidades y categorización estandarizada de los productos existentes.

3. Arquitectura deseada: Odoo como dueño de los datos

La definición de arquitectura que queremos respetar:

  • Odoo es el master de datos. Dueño del catálogo, precios, inventario y de toda integración con terceros.
  • La integración con TecDoc/TecAlliance se hace únicamente hacia Odoo. Odoo consume la API de TecDoc, almacena/cruza lo necesario, y es quien transfiere la información a Shopify.
  • Shopify no se integra directo con TecDoc para datos de catálogo — recibe todo vía Odoo (productos, metafields, compatibilidades).

Punto abierto a discusión (queremos su recomendación): para el buscador por vehículo en la tienda, ¿conviene que el front de Shopify consulte la API de TecDoc en tiempo real (vía un middleware/proxy), o que las compatibilidades se sincronicen como datos a Shopify (metafields/tablas) vía Odoo? Consideraciones que vemos:

  • Tiempo real: datos siempre actualizados, menos volumen a sincronizar, pero agrega dependencia, latencia y consumo de licencia por request en el front.
  • Sincronizado vía Odoo: consistente con la arquitectura "Odoo dueño de todo", funciona aunque TecDoc no responda, pero implica modelar/almacenar el árbol vehículo→pieza y mantenerlo actualizado.
  • Quizá un híbrido: catálogo y compatibilidades sincronizados vía Odoo + autocompletado/búsqueda por OEM en tiempo real.

Esperamos que el plan de trabajo de Augetec incluya esta decisión de arquitectura con su justificación.

4.3 Limitación de la cuenta DEMO (importante para no perder tiempo)

La DEMO viene cargada solo con vehículos comerciales: tipo K = camiones (98,476: IVECO, SCANIA, MB Trucks…) y tipo A = ejes (7,191). Los autos de pasajeros (tipos P/V — lo que vende Embler) están en cero, y los vínculos pieza↔vehículo de las marcas DEMO tampoco están cargados (totalLinkages: 0).

Esto no es un defecto de la API: la estructura, métodos y formato de respuesta son idénticos para todos los tipos de vehículo (lo verificamos con los datos de camiones y ejes, que sí traen modelo-serie, años y atributos técnicos en español). Al contratar el plan con datos Passenger Car para México, los mismos requests regresarán los BMW/Mercedes/Audi. No vamos a pedir más datos a la DEMO; la usamos para validar que la API funciona y conocer la estructura — objetivo cumplido.

4.4 El flujo del buscador por vehículo (ya mapeado contra la API)

1. getManufacturers                  → marcas de auto (mfrId)
2. getLinkageTargets (mfrIds +
   includeVehicleModelSeriesFacets)  → modelos/series con años
3. getLinkageTargets (mfrIds +
   vehicleModelSeriesIds)            → motorizaciones (linkageTargetId)
4. getArticles (linkageTargetId +
   linkageTargetType)                → piezas compatibles con ese vehículo

4.5 Detalles técnicos aprendidos (ahorran tiempo de implementación)

  • Auth solo por header X-Api-Key.
  • País por cuenta: solo mx; mex/am/de devuelven 400.
  • No confundir linkingTargetType (métodos de catálogo de vehículos) con linkageTargetId/linkageTargetType (filtro por vehículo en getArticles).
  • countriesCarSelection es obligatorio en getVehicleIdsByCriteria y getVehicleIdsByKTypeNumber; en este último el parámetro es typeNumber (no kTypeNumber).
  • El legacyArticleId vive en article.genericArticles[].legacyArticleId.
  • En getLinkageTargets, omitir linkageTargetType regresa todo; un tipo sin datos regresa 0 (útil para diagnosticar qué trae una cuenta).
  • Mensajes del servidor: "Field X must be not null" = error de request; "not enabled for this account" = dato/módulo no provisionado. Capas distintas.
  • El endpoint correcto es DLB; existe un DLW con otro formato que no aplica para catálogo.
  • Paginación hasta 10,000 registros por request (perPage/page); perPage: 0 regresa solo conteos y facets.

4.6 Checklist de aceptación para el día 1 del plan de pago

Cuando se active la cuenta de producción, validar en este orden (tenemos los requests listos):

  1. getLinkageTargets tipo V para mx → debe regresar total > 0.
  2. getLinkageTargets con mfrIds=16 (BMW) + facets de series → deben aparecer E46/E90/F30…
  3. Detalle de un vehículo → debe traer ficha de motorización (kW/HP, combustible, años).
  4. El crítico: getArticles con linkageTargetId de ese vehículo → debe regresar artículos > 0 de las marcas contratadas. Este es el único paso del flujo que la DEMO no pudo demostrar de punta a punta (por los linkages en cero) y debe validarse antes de dar por buena la cuenta de producción.

5. Ideas de funcionalidad para el buscador (visión de producto)

  • Selector de vehículo persistente en la tienda ("Mi auto"): el cliente elige una vez Marca → Modelo → Año/Motor y toda la navegación/colecciones se filtra a piezas compatibles. Que se recuerde entre sesiones.
  • Búsqueda por número de parte / OEM con equivalencias: pegar cualquier número (original o de otra marca) y llegar al producto Embler equivalente.
  • Autocompletado en la barra de búsqueda (TecDoc ya lo ofrece vía getAutoCompleteSuggestions).
  • Tabla de compatibilidades en la ficha de producto: "Aplica para: BMW Serie 3 (E90) 320i 2005–2011, …" generada desde TecDoc en lugar de capturarla a mano.
  • Validación de compra: aviso "verificaste que es compatible con tu vehículo" antes del checkout para reducir devoluciones.
  • Enriquecimiento del catálogo existente en Odoo: cruzar nuestros ~12,700 productos contra TecDoc por número de parte para completar OEMs faltantes (~230 productos), especificaciones, y compatibilidades (~985 productos sin datos de año).
  • A futuro: filtro por VIN o por placa (TecAlliance tiene servicios para esto; fuera de alcance inicial).
